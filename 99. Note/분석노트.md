Q1) @Transactional이 같은 클래스 내 호출에서 안 되는 이유는 무엇인가요?
A1) @Transactional은 Spring AOP 프록시 패턴 기반으로 동작합니다.
     @Transactional은 Proxy 객체 메서드 진입 시점에 시작되는데,
     같은 클래스 내 this.method() 호출은 프록시를 거치지 않고 원본 객체를 직접 호출하기 때문입니다. ^e6WDLcrR

Q2) JDK Dynamic Proxy와 CGILB의 차이, Spring Boot는 왜 CGLIB을 기본으로 사용하나요?
A2) Proxy 객체 생성 방식의 차이가 있습니다.
     JDK Dynamic Proxy는 Interface 기반으로 Proxy 객체를 생성하고
     CGLIB은 Class 기반으로 Proxy 객체를 생성합니다.

     CGLIB을 사용하는 이유는 Interface없이 Proxy를 만들 수 있어 제약이 적고
     Interface없이 Class만 생성하는 패턴 등이 많아졌기 때문입니다. ^0UKcuffx

Q3) AspectJ를 쓰면 해결되나요? 실무에서는 잘 안 쓰나요?
A3) AspectJ를 사용하면 Self-Invocation 문제를 해결할 수는 있습니다.
     다만, Compile Time weaving는 빌드 설정이, Load Time Weaving은 JVM 에이전트 설정이 필요해 복잡도가 올라갑니다.
     대부분의 서비스는 클래스 분리만으로 충분히 해결되므로 잘 사용하지 않습니다. ^RvS0MNtU

Q4) 클래스 분리만으로 충분히 해결 가능하다고 하셨는데 분리 기준은 어떻게 잡나요?
A4) 트랜잭션 경계 자체가 곧 하나의 Usecase 단위인 경우가 많습니다.
     DDD나 Hexagonal Architcture에서 Appliation Service Layer 단위가 하나의 Transaction 단위가 되고 내부 도네인 서비스와 분리됩니다.
     이 레이어 경계를 따르면 설계상 자연스러운 분리가 됩니다.
     이러한 상황이라면 클래스 분리가 아니라 이벤트 기반으로 분리가 가능합니다. ^DHqSwEWb

Q5) 이벤트 기반으로 분리 시 Transaction 경계는 어떻게 설정할 수 있을까요?
A5) @EventListenr,                 @TransactionalEventListener
     이벤트 발행 즉시,                 트랜잭션 커밋 후
     트랜잭션에 포함 (같이 롤백)     동작 X
     같은 트랜잭션에 묶고 싶을 때    커밋 확정 후 후처리 ^oBJ3D0vv

Q6) 내부 이벤트가 유실되면 어떻게 처리할 수 있나요?
A6) 내부 이벤트가 유실되면 처리하기 어렵습니다.
     이벤트 유실을 방지하기 위해 Transactional Outbox Pattern 도입 고려가 필요할 수 있습니다.
     결제 완료 알림, 감사 로그, 외부 시스템 연동 등 정합성 보장이 반드시 필요한 경우에 도입할 수 있습니다. ^TyM3fpLM
