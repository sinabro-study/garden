# 📌 서비스 컨셉

### 서비스명 (예시)
광고·리워드 기반 커머스 플랫폼
- 사용자는 가입하면 포인트를 받고
- 출석 체크 / 이벤트 참여 / 광고 시청 등으로 포인트를 모음
- 포인트로 상품 구매 가능

---
# 1️⃣ 기본 단계 – Monolithic (서비스 초기)

## 🧾 비즈니스 상황
- 이제 막 런칭한 스타트업
- MAU 1,000명 이하
- 트래픽 예측 불가
- 빠른 개발이 중요
- 인프라 비용 최소화 필요

## 📌 요구사항
- 회원가입 시 1,000 포인트 지급
- 포인트는 사용자 잔액에 누적
- 트랜잭션 일관성 중요 (가입 실패하면 포인트도 지급 X)

## 🏗 기술 설계

### 구조
- Monolithic Spring Boot
- 단일 DB (MySQL)
- 단일 트랜잭션

`UserService   └─ createUser()         └─ PointService.giveSignupPoint()`

### 특징
- 동일 트랜잭션 내 처리
- 강한 일관성 (Strong Consistency)
- 개발 속도 빠름
- 단일 장애점(SPOF) 존재

### 포트폴리오에서 강조할 포인트
- "왜 MSA로 시작하지 않았는가?"
- 비용 대비 효율
- YAGNI 철학 적용
- 비즈니스 성장 단계에 따른 아키텍처 선택

---

# 2️⃣ 중간 단계 – 트래픽 증가 → MSA 도입

## 🧾 비즈니스 상황
- 출석 체크 이벤트 시작
- 하루 수만 건의 포인트 적립 요청
- 회원가입은 하루 수백 건 수준
- 포인트 서비스 CPU 사용량 급증
- API 응답 지연 발생

## 📌 새로운 요구사항
- 출석 체크 시 50 포인트 지급
- 포인트 집계는 정확해야 함
- 비용 절감 위해 필요한 서비스만 Scale Out

---

## 🏗 아키텍처 변경

### 서비스 분리
- User Service
- Point Service

### 통신 방식
- 동기식 REST 통신
- OpenFeign or WebClient

`User Service → (HTTP) → Point Service`

### DB 분리
- user_db
- point_db

---

## 🔥 왜 동기식인가?
- 회원가입 트래픽 적음
- 즉시 포인트 지급 필요
- 구조 단순 유지
- 운영 복잡도 최소화

---

## 📌 발생 가능한 문제
- Point Service 장애 시 회원가입도 실패
- 네트워크 Latency 증가
- 분산 트랜잭션 문제

---

## 🎯 여기서 어필 포인트
- MSA는 무조건 Event 기반이 아님
- 트래픽 패턴 기반 설계
- 비용 최적화 전략
- 서비스별 Scale Out

---

# 3️⃣ 고급 단계 – 폭발적 성장

## 🧾 비즈니스 상황

- 대형 광고 집행    
- 하루 회원가입 수만 건
- 동시성 폭증
- Point Service에 병목 발생
- 응답 대기 시간 증가
- Timeout 증가

---

## 🚨 문제

동기식 구조:
`회원가입 → Point API 호출 → Point 처리 완료 후 응답`

Point가 느리면:
- User Service Thread Block
- 커넥션 풀 고갈
- 전체 서비스 장애

---

# 4️⃣ 개선 전략 1 – Circuit Breaker 도입

예:
- Resilience4j

### 결과
- Point 장애 시 빠르게 실패
- User는 가입 성공
- 포인트는 미지급

하지만...

👉 비즈니스적으로 문제  
"회원가입했는데 포인트 안 들어왔어요"

---

# 5️⃣ 최종 진화 – Event 기반 아키텍처

## 📌 변경 전략

### 흐름

`User 가입 성공    ↓ UserSignedUpEvent 발행    ↓ Kafka    ↓ Point Service 소비    ↓ 포인트 지급`

---

## 🏗 기술 구성
- Kafka
- Event Driven Architecture
- 비동기 처리
- Eventually Consistency

---

## 📌 장점
- User Service 응답 빠름
- Point 장애와 분리
- Scale 독립적
- 트래픽 버퍼링 가능

---

## 📌 해결해야 할 문제

- 중복 이벤트 처리 → Idempotency Key    
- 보상 트랜잭션
- 이벤트 유실 방지
- DLQ 처리
- Saga 패턴 고려

---

# 🔥 포트폴리오에서 이렇게 말하면 좋음

> 우리는 처음부터 MSA를 선택하지 않았다.  
> 트래픽과 비즈니스 상황을 기반으로 점진적으로 아키텍처를 진화시켰다.
